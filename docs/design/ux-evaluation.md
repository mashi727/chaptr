# Chaptr UX 評価（PoC ベース）

最終更新: 2026-06-25
対象: `poc/web/`（Web Audio PoC）／ 中核タスク = リハーサル動画 → 精密チャプター打点 → 書き出し

## 0. 評価方法

- **実機相当のビューポートで PoC を自動操作し、計測 + スクリーンショット取得**（Playwright, deviceScaleFactor=2, touch有効）。
  - iPhone 縦 390×844 / iPad 縦 834×1112 / iPad 横 1194×834
- 合成チャープを読み込み、チャプター2点を打った「実利用に近い状態」で計測。
- 観点: ①レイアウト占有率 ②タッチターゲット ③中核タスク（精密打点）のヒューリスティック評価 ④デバイス役割 ⑤フレームワーク選定への含意。

### 計測結果（実測値）

| 指標 | iPhone縦 | iPad縦 | iPad横 | 基準/評価 |
|---|---:|---:|---:|---|
| ヘッダ占有率 | **45%** | 18% | 21% | iPhoneで過大（操作部が画面の半分） |
| 波形+スペクトロ占有率 | **38%** | 69% | 62% | iPhoneで内容が圧迫 |
| 最小ボタン高 | 29px | 29px | 29px | **Apple HIG 44px 未満（17/17ボタン）** |
| 横はみ出し | なし | なし | なし | 折返しで回避（良）|
| チャプター一覧 | 見切れ | 可 | 可 | iPhoneで下部が隠れる |

> 結論の先取り: **iPad/デスクトップは現状でも実用域。iPhone 縦は精密編集UIとして窮屈**で、
> 役割を分ける設計（§4）が妥当。加えて全機種共通で「精密打点を支える操作系」が不足（§3）。

## 1. 良い点（活かす）

- **波形＋スペクトログラムの同時表示**は、本ドメイン（楽曲境界＝音の立ち上がり）に的確。
  スペクトログラムで曲の切れ目・休符・テンポ変化が視認でき、波形でアタックが分かる。iPad横で特に良好。
- **ms 読み出しが常時可視**（`00:00.300` ＋ `300.0 ms`）。システム状態の可視性が高い。
- **ナッジ（±10/±100ms）** が最初から在る。タッチでの粗い操作を ms 精度に追い込める基本線が通っている。
- **クリック=シーク / ドラッグ=スクラブ**は素直。波形上で直接位置を触れる。
- 共有タイムラインのズーム/スクロールで、可視範囲STFTが即時再計算され体感が軽い（PoC検証済）。

## 2. 重大度つき所見（ヒューリスティック）

凡例: **P0=中核体験を妨げる / P1=満足度を大きく損なう / P2=磨き込み**

| # | 重大度 | 所見（根拠） | 推奨 |
|---|:--:|---|---|
| F1 | **P0** | iPhone縦でヘッダ45%占有・内容38%・一覧見切れ（実測）。精密編集が成立しない。 | レスポンシブ化。phoneは操作部を最小化（再生＋ナッジ＋打点のみ）＋単一ビュー切替。§4の役割分担。 |
| F2 | **P0** | 再生中に**ビューが再生位置を追従しない**。ズーム時に再生ヘッドが画面外へ。「聴きながら打点」が破綻。 | 再生中オートスクロール（ヘッド固定 or 端で送り）。トグルで切替。 |
| F3 | **P0** | **オンセット/トランジェントへのスナップが無い**。楽曲境界＝立ち上がりなのに、目視＋手動で合わせる必要。 | 既存ピークデータを使い「最近接ピーク/立ち上がりへスナップ」。精度と速度を同時に底上げ。 |
| F4 | P1 | **タッチターゲット29px < 44px**（17/17）。指で誤タップしやすい。 | 主要操作を44px以上。phoneは大型トランスポート。 |
| F5 | P1 | **ピンチズーム未実装**（UIヒントは「ピンチ」と表示するが実体はホイールのみ）。iPadはボタン頼み。 | 2本指ピンチ=ズーム / 1本指パン=スクロール を実装。タッチの基本作法に合わせる。 |
| F6 | P1 | **時間ルーラー/目盛りが無い**。可視範囲の絶対時刻・スケールが読めない（再生ヘッドの数値のみ）。 | 上端に時間軸ルーラー＋現在ズーム表示。全体位置を示すミニマップ/スクロールバー。 |
| F7 | P1 | **取り消し（Undo）が無い**。打点削除は確認なしで即時。誤操作の復帰不可。 | Undo/Redo（最低でもチャプター操作）。削除はスワイプ＋取り消しトースト。 |
| F8 | P1 | **デスクトップのキーボード操作が皆無**（Space/矢印/M等）。熟練ワークフローが遅い。 | Space=再生、←→=ナッジ、Shift+←→=粗送り、M=打点、Enter=改名。 |
| F9 | P2 | ジェスチャ（クリック/ドラッグ/ホイール）が**非可視で発見不能**。 | 初回ヒント or 操作チップ。アイコンで示唆。 |
| F10 | P2 | **打点プレビュー（前後ループ試聴）が無い**。置いた位置の良し悪しを確認しづらい。 | 「マーカー前後2秒をループ試聴」。境界の追い込みに有効。 |
| F11 | P2 | タイトルバーが冗長（2行折返し）で縦を消費。 | アプリ名のみ＋情報はステータス行へ集約。 |
| F12 | P2 | スペクトログラムに**周波数軸の目盛りが無い**。 | 縦軸に主要周波数ガイド（任意）。 |

## 3. 中核タスクのUX：精密チャプター打点（最重要）

このツールの価値は「**正しい ms に境界を置けること**」。タッチでは指の太さが精度の敵になる。
タスクを「①おおまかに移動 → ②微調整 → ③確定 → ④検証」の4段に分解し、各段に専用の支援を置く。

| 段 | 現状 | 目標UX |
|---|---|---|
| ① 粗移動 | 波形クリックでシーク（粗い） | クリック/タップで近傍へ。ミニマップで全体把握（F6）。 |
| ② 微調整 | ナッジ±10/±100ms（有） | **ジョグ/シャトル帯**（ドラッグで速度可変送り）＋**ピンチで拡大**（F5）して波形を細かく。 |
| ③ 確定 | 再生位置へ打点（有） | **オンセットスナップ**（F3）で機械的に立ち上がりへ吸着。確定時に微ズレを自動補正。 |
| ④ 検証 | なし | **前後ループ試聴**（F10）＋再生ヘッド追従（F2）で「聴いて確かめる」。 |

> 設計含意: ②③は **既存の min-max ピーク / STFT データを再利用**して実装できる（新規DSP不要）。
> 「ピークの極大」や「短時間エネルギーの急増点」を候補にスナップするだけで、タッチ精度問題の大半が解ける。

## 4. デバイス役割の分担（推奨）

iPhone縦で精密編集を完全提供しようとすると窮屈（F1）。**機種ごとに主目的を変える**のが現実解。

| デバイス | 主目的 | UI方針 |
|---|---|---|
| **デスクトップ / iPad横** | **精密編集**（境界の追い込み・命名・書き出し） | 波形＋スペクトロ大画面、キーボード/ポインタ、ジョグ。 |
| **iPad縦** | 編集（やや簡略） | 同上だが操作部をやや圧縮、単一ビュー切替可。 |
| **iPhone** | **レビュー / 速記キャプチャ / 仕上げ確認** | 大型トランスポート＋ナッジ＋打点に集中。波形/スペクトロは切替式の単一表示。一覧は別ペイン。 |

`*.vce.json` が可搬なので「iPhoneで仮打ち → iPad/PCで追い込み」という往復が自然に成立する（設計書§5）。

## 5. フレームワーク選定への含意（設計書§3へのフィードバック）

UX観点は Flutter vs Tauri の判断材料を一つ増やす:

- 本ツールは **ピンチ/パン/慣性スクロール/120Hz相当の滑らかな再描画** が体験品質を左右する（F2,F5）。
- **Flutter（Skia/Impeller）**: ジェスチャ系・高頻度カスタム描画・タッチ慣性が安定。本用途の体験で有利。
- **Tauri（Webview）**: Rust DSPは強い一方、**iOS WKWebView 上での canvas ピンチズーム/慣性スクロールの作り込み**が必要で、体験の詰めにコストがかかる可能性。
- **Swift ネイティブ**: タッチ慣性・ハプティクス・ProMotion 追従は最良。Apple集中なら体験面で頭一つ抜ける。

> よって PoC-3（実機検証）では、機能の成否だけでなく **「ピンチズーム/再生ヘッド追従/オンセットスナップ」3点の操作感**を
> 採用候補フレームワークで触って比較することを推奨。Flutter vs Tauri の最終判断はここで付く。
